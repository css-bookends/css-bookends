# CSS-Bookends Gameplan

The agreed sequence for splitting the work into two packages and getting both
out the door. This is the order of operations; `MONOREPO_PLAN.md` has the deeper
structural detail and `sessions.md` has the running history.

## The vision in one line

Two packages, two audiences.

- **css-calipers**: pure measurement, going to **v1**. Stable, focused, unscoped
  on npm. The people who only want typed measurements live here and nothing
  breaks for them.
- **css-bookends**: the umbrella monorepo, **experimental v0.x**. Holds
  everything (calipers as one book, plus media queries and all the portfolio
  helpers). Pick and choose the "books" you want.

### Lexicons and books (working model, may change)

The point is to distinguish the **foundational** pieces from the **standalone
libraries** built on them. The metaphor: a lexicon is a vocabulary; books are
written using lexicons.

- **Lexicons** are the foundational vocabularies (`calipers`, `colours`,
  `spacing`).
- **Books** are the standalone libraries (`margins`, `borders`, `shadow`, …),
  each built on one or more lexicons.

Every piece is installable on its own and pulls in only what it actually depends
on. Beyond that, the taxonomy is a working model we will refine as the helpers
come over; not worth hardening now.

## End state we are building toward

- `css-bookends` is public, a pnpm monorepo, source of truth for all books.
- `packages/calipers` is **pure measurement only**: `m()`, the unit helpers,
  `ratio`. No media queries. It publishes to npm under the unscoped name
  `css-calipers`.
- Every other lexicon and book (colours, spacing, media queries, borders,
  margins, shadow, ...) publishes under `@css-bookends/*`, all on their own `0.x`
  line. Internal dependencies are wired with `workspace:*` (most depend on
  `css-calipers`, but not all).
- The standalone `slafleche/css-calipers` repo becomes a **read-only mirror** of
  `packages/calipers`, regenerated by the publish/mirror workflow. It carries the
  v1 git tag and the new README.

## The sequence (order matters)

### 1. Wipe bookends, copy css-calipers in clean

Start the monorepo fresh rather than building on the current subtree scaffold.
Bring css-calipers in as `packages/calipers`, keeping its npm name `css-calipers`
and its version line. This copy is the source of truth from here on.

### 2. Copy the other helpers in as books

Each helper lands at `packages/<book>` and publishes as `@css-bookends/<book>`,
all `0.x`, each declaring `css-calipers` via `workspace:*`.

- **Media queries** comes out of calipers and becomes
  `@css-bookends/media-queries`. This is real surgery, not a file move, because
  calipers' factory is coupled to it:
  - Delete `packages/calipers/src/internal/createMediaQueriesApi.ts` and strip
    the media-queries wiring out of `packages/calipers/src/factory.ts` (drop the
    `.mediaQueries` namespace from `createCalipers()`).
  - Remove the `./mediaQueries` export, the media-queries build/test scripts, the
    `tests/runtime/mediaQueries/**` and media-queries type tests, and the media
    queries section of the calipers README.
  - Move `src/mediaQueries/**`, its tests, and `README_MEDIAQUERIES.md` into the
    new `packages/media-queries` book.
- **Portfolio helpers** follow, sourced from
  `portfolio/src/styles/helpers/*.helper.ts` (colorWrap, borders, shadow,
  spacing, and the rest). Bring each one's tests over with it. Spacing is the
  first 1.x candidate but still needs work; the rest stay `0.x`.

`ratio` stays in calipers. It is a measurement primitive and media queries
depends on its type.

### 3. The clean-split gate (write the tests here)

Prove that **css-calipers can be split off with nothing else attached**. This is
the gate that has to go green before anything ships:

- `packages/calipers` has zero dependencies on any other book (no `workspace:*`
  pointing inward, no stray imports from `../media-queries` etc.).
- It builds, tests, and type-checks standalone: `pnpm --filter css-calipers test`
  is green and there are zero `mediaQueries` references left in the package.
- The mirror split of `packages/calipers` produces a self-contained, installable
  package (dry-run the split, install the output, run its tests).

### 4. Release bookends v0.0 (experimental)

Once the split gate is green, cut the first umbrella release. Bookends is marked
experimental at `0.x`. Books publish independently via Changesets.

### 5. Mirror css-calipers out and ship v1

Update the publish flow so it mirrors `packages/calipers` to the standalone
`slafleche/css-calipers` repo. That repo:

- loses media queries (now pure measurement),
- gains a new README with the migration note (below),
- gets the **v1** tag and npm publish under `css-calipers`.

Then we're cookin'.

## The v1 migration note (goes in the css-calipers README)

> If you only use CSS-Calipers (measurement), you can update to v1 as-is; nothing
> breaks. If you used the media queries, they (along with the other experimental
> features) now live in CSS-Bookends. In that case you are better off switching to
> CSS-Bookends: it contains everything, and you pick and choose the "books" (the
> parts) you want.

## Decisions locked

- Monorepo is the single source of truth. Standalone repos are read-only mirrors.
- `css-calipers` stays **unscoped** on npm (protects existing downloads, the npm
  page, badges, shared links). Everything else is `@css-bookends/*`.
- Media queries leaves calipers **before** v1, so v1 ships pure. Clean break, no
  deprecation period: bookends is private today and there are no published
  consumers of the `css-calipers/mediaQueries` subpath to protect.
- Bookends ships **first** (v0.0), calipers v1 mirror goes out **after**.
- pnpm workspaces + Changesets for independent per-book versioning and publish.
- Mirror via `symplify/monorepo-split-github-action` (per `MONOREPO_PLAN.md`).
